## 1. 产品概述

### 1.1 产品愿景

> **AI让这一切变可能**
>
> 一款轻量化低门槛的ASD儿童家庭干预辅助工具，让中轻度ASD孩子获得持续有效的家庭干预资源，同时不断鼓励家长持续进行家庭干预。

### 1.2 核心价值主张

> **让每一个情境问题，都能转化成为一次有效干预**

### 1.3 产品定位

**定位**：ASD儿童家庭干预辅助工具

**目标用户**：
- 主要用户：2-6岁中轻度ASD儿童的家长（主要是妈妈）
- 次要用户：孩子本身（桌面干预，第二阶段）

**传播方式**：家长自主口口相传

### 1.4 产品原则（Daniel底线，不可违背）

1. **产品安全**：做ASD儿童干预产品，绝对不能负面影响孩子的干预效果。这是最高优先级，高于产品优化和商业目标。
2. **追求有效**：产品不追求"有趣"或"低门槛"，追求专业、有效、不制造焦虑。
3. **不做空洞鼓励**：不做空洞正向鼓励，不给家长制造焦虑。
4. **MVP优先**：砍功能不手软。

### 1.5 品牌调性

- **温暖但不幼稚**：专业但不高冷，禁止卡通化
- **不追求低门槛**：追求专业，有效、不制造焦虑
- **色彩体系**：
  - 背景色：#F7F4F0
  - 卡片色：#FFFFFF
  - 强调色：#7CB342
  - 文字主色：#2D2D2D
  - 文字次色：#6B6B6B

### 1.6 有效干预的定义

> 减少问题行为，让孩子更融入社交场合或社会规则中。能适应社会规则，能独立自主照顾自己，能表达自己的想法。

| 时间状态 | 碎片化时间，每次5-12分钟 |

| 维度 | 描述 |
|------|------|

```

干预经历：已接受或正在接受机构干预
经济压力：干预费用3000-8000元/月，占家庭支出30-50%
```

### 2.3 家长心理状态分层

| 阶段 | 心理特征 | 典型行为 |
|------|----------|----------|

| **接受期** | 理性、寻求支持 | 寻找可持续方案 |
| **长期抗战** | 疲惫但坚持 | 寻找平衡、家庭压力 |

### 2.4 核心用户痛点

| 痛点 | 描述 | 对应解决方案 |
|------|------|-------------|
| **无法持续坚持** | 长时间看不到效果、方案过于专业复杂、门槛过高、家长忙于工作 | 目标拆解、任务拆分、每日5-12分钟 |
| **不知道怎么做** | 没有专业干预知识、没有轻松低门槛的指导 | AI语义分析匹配任务包、步骤引导 |
| **无法感知进步** | 无法及时获得反馈、效果难以显现、无法清晰对比 | 微感知系统 |

## 3. 核心痛点

### 3.1 痛点一：不知道"现在该做什么"

**问题描述**：
- 每天面对真实情境（穿衣、洗澡、外出）
- 缺乏具体行动指导
- 机构教的和家里发生的脱节

**家长原声**：
> "看了一堆干预理论，真到生活中还是不知道怎么做"

**解决方案**：
- 以家长遇到的**真实问题**为入口
- AI语义分析，匹配最合适的情境任务包
- 每天3-5步，5-12分钟

### 3.2 痛点二：做了不知道有没有用

**问题描述**：
- 没有系统反馈机制
- 无法判断方向是否正确
- 看不到进步的"小信号"

**家长原声**：
> "不是不想坚持，是真的不知道每天该做什么"

**解决方案**：
- **微感知系统**：让家长"看到"微小进步
- 每日反馈记录（必要+可选字段）
- 里程碑达成展示

### 3.3 痛点三：很难坚持

**问题描述**：
- 时间和精力被碎片化消耗
- 一个人面对，孤独感强
- 家庭成员配合不一致

**家长原声**：
> "孩子哭没哭好像就是唯一的标准"

**解决方案**：
- 即时微感知（每次训练后）
- 里程碑微感知（每7天）
- 历史记录回顾

## 5. 微感知系统设计

> **核心理念**：帮助非专业家长识别"微小进步"
>
> **理论基础**：福格行为模型 (B=MAP)
> - **动机（Motivation）**：让家长看到微小进步，强化"有效"认知
> - **能力（Ability）**：降低专业门槛，翻译专业术语
> - **提示（Prompt）**：触发家长行动的信号

### 5.1 第一层：即时微感知（每次训练后）

**目标**：训练完成后，立即让家长"感知"到孩子的微小进步

**设计原则**：
- 不使用专业术语
- 使用对比句式（"比昨天..."）
- 正面描述为主
- 控制在20字以内

**文案风格**：温暖简洁为主（B），可量化（C）为辅

**专业信号 → 家长语言对照表**：

| 专业信号 | 家长能理解的语言 |
|----------|-----------------|
| "呼名反应延迟减少0.5秒" | "孩子对名字反应更快了" |
| "目光接触时长增加2秒" | "孩子看你的时间变长了" |
| "P3辅助降到P1辅助" | "孩子更独立了" |
| "目标行为出现率60%→75%" | "今天有进步，继续加油" |

### 5.2 第二层：里程碑微感知（每7天）

**目标**：让家长看到阶段性的技能习得

**触发频率**：固定每7天（与任务包周期对齐）

**里程碑类型**：

| 类型 | 示例 | 展示形式 |
|------|------|---------|
| 技能习得 | "从抗拒刷牙 → 能拿起牙刷" | 成就卡片 |
| 辅助撤除 | "从全辅助 → 只需口头提示" | 进度条 |
| 泛化进展 | "在超市也能做到了" | 场景标签 |

**成就卡片设计**：
```
┌─────────────────────────────────┐
│  🎉 里程碑达成！                  │
│                                 │
│  第7天 · 穿衣抗拒               │
│                                 │
│  孩子今天能够：                  │
│  ✓ 接受家长触碰衣物              │
│  ✓ 主动伸手进袖子（少量辅助）     │
│                                 │
│  相比第1天：进步明显！           │
│                                 │
│  [分享到微信] [继续第2周]        │
└─────────────────────────────────┘
```

### 5.3 第三层：长期微感知（每月复盘）

**目标**：让家长看到持续干预的价值

**状态**：后续版本独立需求，MVP暂不做

**月度复盘维度**：

| 维度 | 指标 | 展示形式 |
|------|------|---------|
| 训练坚持 | 连续训练天数 | 连续打卡墙 |
| 技能进展 | 累计掌握的技能数 | 技能图谱 |
| 行为变化 | 问题行为出现频率 | 趋势图 |
| 家长信心 | 主观信心评分 | 信心指数 |

### 5.4 微感知系统技术实现

**数据采集**：
```javascript
// 每次训练的反馈数据
{
  scenario_id: "xxx",
  day: 3,
  feedback_rating: "good", // good/neutral/bad
  train_count: "3~5次",

  // 微感知专有字段
  progress_signals: {
    gaze_duration_change: "+2秒",
    prompt_level_change: "P3→P1",
    independence_score: 75,
  },
  milestone_unlocked: null,
  micro_perception_text: "孩子对名字反应更快了",
}
```

**AI生成微感知文案**：
```python
# 微感知文案 Prompt
微感知文案 = AI_Translate(
  专业信号 = feedback_data.progress_signals,
  家长语言风格 = "温暖、简洁、正向",
  字数限制 = 20
)
```

## 9. AI匹配系统设计

### 9.1 匹配流程

```
家长输入（文字/语音）
       ↓
语义分析 → 情境识别
       ↓
匹配任务包（单包匹配 + 无匹配记录）
       ↓
返回匹配结果
```

### 9.2 MVP匹配策略

**方案B：单包匹配 + 兜底**

| 场景 | 处理方式 |
|------|---------|
| 高置信匹配 | 直接返回唯一任务包 |
| 无匹配 | 记录输入，启动内容运营流程 |

**匹配逻辑**：
- 基于语义分析识别情境类型
- 匹配对应场景的任务包
- 无匹配时记录数据，供内容运营使用

### 9.3 V1.x 匹配置信度设计

**方案A：置信度分级**（后续版本）

| 置信度 | 处理方式 |
|--------|---------|
| 高置信（>80%） | 直接返回唯一任务包 |
| 中置信（50-80%） | 返回2-3个候选，用户选择 |
| 低置信（<50%） | 返回多候选 + 提示"请选择最符合的" |

**触发条件**：基于MVP阶段收集的匹配数据决定是否升级

### 9.4 无匹配处理流程

```
用户输入无法匹配
       ↓
记录到数据库
（输入文本 + 语义分析结果）
       ↓
内容运营团队看到
       ↓
按任务包框架生产新内容
       ↓
上线覆盖该场景
```

## 10. 核心业务流程

### 10.1 主流程：情境 → 训练 → 反馈

```
┌─────────────────────────────────────────────────────────────┐
│                        用户旅程                              │
└─────────────────────────────────────────────────────────────┘

[添加情境] → [AI匹配方案] → [确认方案] → [每日训练] → [记录反馈] → [追踪进度]
     │              │              │            │            │            │
     ↓              ↓              ↓            ↓            ↓            ↓
  描述问题      匹配任务包      查看目标     执行步骤     选择效果     查看进度
  选择标签      返回结果        开始训练     记录次数     提交反馈     微感知
```

### 10.2 关键路径（Happy Path）

```
1. 用户打开App → 首页
2. 输入文字描述或选择快捷场景
3. AI匹配任务包
4. 查看方案概览
5. 点击"开始第1天训练"
6. 按步骤执行训练
7. 点击"我完成了今天的训练"
8. 选择效果 + 次数
9. 提交反馈
10. 返回首页，查看即时微感知
```

### 10.3 决策点

| 节点 | 决策 | 路径 |
|------|------|------|
| 首页 | 是否有情境？ | 有 → 查看列表；无 → 添加情境 |
| input.html | 输入是否为空？ | 空 → 提示；有效 → 匹配 |
| 匹配结果 | 是否有匹配？ | 有 → 查看详情；无 → 记录反馈 |
| training.html | 是否完成训练？ | 完成 → 反馈页；退出 → 保持进度 |
| feedback.html | 是否提交？ | 提交 → 首页微感知；取消 → 训练页 |

### 10.4 状态机

**任务包状态流转**：
```
[创建] → active（进行中）
           ↓
    ┌──────┼──────┐
    ↓      ↓      ↓
completed abandoned deleted
 (已完成) (已放弃) (已删除)
```

**每日任务状态**：
```
[当日开始] → pending
                ↓
    ┌───────────┼───────────┐
    ↓           ↓           ↓

```

## 18. 多角色评审结论

> 评审日期：2026-05-12
> 评审角色：产品经理、技术架构师、UX研究员

### 18.1 三角色共同指出的高优先级问题

| 优先级 | 问题 | 影响 |
|--------|------|------|
| **P0** | 即时微感知文案缺失 | 产品核心价值"感知微小进步"无法传达 |
| **P0** | Demo版数据无持久化 | 用户7天投入随时清零 |
| **P0** | AI匹配只用10包非66包 | 无法验证66包匹配质量 |

### 18.2 各角色核心建议

**产品视角**：
- 推送提醒：可用"柔性提醒"替代（微信服务号周报）
- 置信度演进：需提前埋点，否则V1.x无限延期
- 66包前台：需分层（5-8个P0核心 + 更多折叠）

**技术视角**：
- 立即：统一match_package_with_ai()使用完整50-66包索引
- Demo验证：引入anonymous UUID隔离数据
- MVP：手机号登录简化方案（SQLite存验证码+HMAC token）

**UX视角**：
- 反馈后展示微感知文案（停留2-3秒）
- 首页增加"最近进步速览"
- 语音输入必须实现完整链路

### 18.3 Daniel决策结果

| 决策点 | 选择 | 说明 |
|--------|------|------|
| 微感知文案生成方式 | **A** | 硬编码扩充到50条，模板+变量，提升丰富度 |
| Demo版数据持久化 | **B** | anonymous UUID + localStorage |
| 语音输入 | **B** | Demo版不做语音，MVP再做 |

### 18.4 Demo版更新设计决策

**Demo版含以下功能**：

| 模块 | 功能 | 说明 |
|------|------|------|
| 数据持久化 | anonymous UUID + localStorage | 防止数据清零 |
| 微感知 | 硬编码50条模板文案 | 扩充21条，提升体验 |
| 语音 | 不做 | 专注核心闭环 |
